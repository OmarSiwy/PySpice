User Guide

Use DeSpice from Python to describe circuits, produce SPICE netlists, select simulator backends, run analyses, read waveforms and measurements, lint designs, and reuse ready-made analog and mixed-signal testbenches.

What It Is

DeSpice is a Rust-powered Python package named pyspice-rs. It gives Python users a circuit-building API that emits SPICE netlists and can run simulator backends such as ngspice, Xyce, LTspice, Spectre, and VacaSk.

The package is useful when you want Python to be the source of truth for analog or mixed-signal experiments: generate circuits, configure analyses, lint before running, switch backends, parse results into Python objects, and create repeatable validation flows.

Primary features

  • Python API for resistors, capacitors, inductors, sources, semiconductors, switches, transmission lines, XSPICE elements, Verilog-A, Verilog blocks, and raw SPICE.
  • Unit helpers using Python's @ operator, for example 10 @ u_V and 2 @ u_kOhm.
  • Operating point, DC, AC, transient, noise, transfer function, sensitivity, pole-zero, distortion, Fourier, RF, S-parameter, harmonic balance, and statistical analysis entry points.
  • Backend detection and feature-aware routing.
  • Reusable design testbench recipes for amplifiers, converters, switches, PLLs, sample-and-hold blocks, and bandgap references.

Outputs

  • Human-readable netlists via str(circuit) or testbench.netlist("ngspice").
  • Simulation result objects with node lookup, branch lookup, sweep axes, frequency axes, time axes, and parsed .meas values.
  • JSON-serializable intermediate representation for structured circuit and testbench storage.
  • C ABI build mode for non-Python consumers.

Install

The project is built with Rust, PyO3, maturin, and Python 3.10 or newer. You also need at least one simulator backend installed if you want to run simulations instead of only generating netlists.

Recommended development shell

source .venv/bin/activate
pip install maturin numpy pytest
maturin develop

Quickstart

This voltage divider shows the standard workflow: create a circuit, add parts, print the netlist, create a simulator, run an analysis, and read a node voltage.

from pyspice_rs import Circuit
from pyspice_rs.unit import u_V, u_kOhm

circuit = Circuit("Voltage Divider")
circuit.V(name="in", positive="input", negative=circuit.gnd, value=10 @ u_V)
circuit.R(name="1", positive="input", negative="output", value=2 @ u_kOhm)
circuit.R(name="2", positive="output", negative=circuit.gnd, value=1 @ u_kOhm)

print(str(circuit))

sim = circuit.simulator(simulator="ngspice")
op = sim.operating_point()
print(f"V(output) = {op['output']:.3f} V")

The emitted netlist is ordinary SPICE:

.title Voltage Divider
Vin input 0 10
R1 input output 2k
R2 output 0 1k
.end

Mental Model

ConceptMeaningHow you use it
CircuitA top-level SPICE deck built from Python calls.circuit = Circuit("name"), then add parts.
NodeAn electrical connection point.Use strings like "vin", "vout", and circuit.gnd.
ElementA SPICE component line with a prefix such as R, C, L, V, M, or X.Call circuit.R(...), circuit.MOSFET(...), and so on.
SimulatorConfiguration and execution wrapper around a circuit.sim = circuit.simulator(), then call analysis methods.
BackendThe executable simulator that runs the netlist.Auto-detect or force "ngspice", "xyce", "ltspice", "spectre", or "vacask".
TestbenchA structured DUT wrapper with stimulus, analyses, saves, measurements, corners, and backend-specific netlist emission.Use Subcircuit plus Testbench or packaged recipes from testbenches.

Ground is always node "0". The convenience property circuit.gnd returns that string. Components that share the same node string are electrically connected.

Access And Mutation

print(circuit["load"])              # lookup an element by name

circuit.raw_spice(".options reltol=1e-6")
circuit.options(RELTOL="1e-4", METHOD="gear")

Models And Subcircuits

Device models

circuit.model("nch", "NMOS", LEVEL=1, VTO=0.4, KP=200e-6)
circuit.model("pch", "PMOS", LEVEL=1, VTO=-0.4, KP=100e-6)
circuit.model("sw1", "SW", VT=2.5, VH=0.5, RON=1, ROFF=1e6)

Common model kinds include D, NPN, PNP, NMOS, PMOS, NJF, PJF, NMF, SW, and CSW.

External model files

circuit.include("/path/to/device_models.spice")
circuit.lib("/path/to/corners.lib", "tt")
circuit.parameter("Rval", "1k")
circuit.R(name="param", positive="a", negative="b", value=0, raw_spice="{Rval}")

Subcircuit instances

For quick pass-through, define raw .subckt text and instantiate it with X. For structured reusable designs, prefer Subcircuit and Testbench as shown later.

circuit.raw_spice(".subckt inverter in out vdd vss")
circuit.raw_spice("Mn out in vss vss nch")
circuit.raw_spice("Mp out in vdd vdd pch")
circuit.raw_spice(".ends inverter")
circuit.X("1", "inverter", "in", "out", "vdd", "0")

Simulation

Create and configure a simulator

sim = circuit.simulator()                    # auto-detect backend
sim = circuit.simulator(simulator="xyce")    # force a backend

sim.options(RELTOL="1e-4", ABSTOL="1e-12")
sim.temperature = 27.0
sim.nominal_temperature = 27.0
sim.initial_condition(out=0.0)
sim.node_set(bias=0.7)
sim.save("V(out)", "V(in)")
sim.save_currents = True
sim.measure("tran", "trise", "TRIG", "V(out)", "VAL=0.5", "RISE=1", "TARG", "V(out)", "VAL=4.5", "RISE=1")

Results

Result objects are Python wrappers around parsed simulator output. Use index access for explicit lookup and convenience properties for axes.

op = sim.operating_point()
print(op["vout"])
print(op.measures)

dc = sim.dc(Vin=slice(0, 1.8, 0.01))
print(dc.sweep)
print(dc["out"])
print(dc.nodes)
print(dc.branches)

ac = sim.ac("dec", 100, 1, 1e9)
print(ac.frequency)
gain = ac["out"]

tran = sim.transient(1e-9, 100e-9)
print(tran.time)
print(tran["out"])

Most waveform-like result types support result["name"], attribute-style lookup for valid Python identifiers, and a measures dictionary for parsed scalar measurements. XyceFftAnalysis also exposes frequency, magnitude, phase, enob, sfdr_db, snr_db, and thd_db.

Backends

Use CircuitSimulator.available_backends() to see what is installed. Auto-detection also considers requested analysis type and required circuit features.

from pyspice_rs import CircuitSimulator
print(CircuitSimulator.available_backends())

sim = circuit.simulator()
print(sim.check_backend("ngspice"))

Feature-aware selection: XSPICE forces ngspice. OSDI requires ngspice, VacaSk, or Spectre. .control blocks require ngspice. Verilog co-simulation currently routes to ngspice or Spectre. If features conflict, auto-selection reports the incompatibility instead of silently picking an invalid backend.

Linting

The linter checks netlist structure and backend-specific hazards before you spend time on a failing simulator run.

from pyspice_rs import lint

report = lint("""
.title lint demo
V1 in 0 1
R1 in out 1k
.end
""", backend="ngspice")

for error in report["errors"]:
    print("error:", error["message"])
for warning in report["warnings"]:
    print("warning:", warning["message"])

Checks include missing .end, missing ground, duplicate elements, floating nodes, zero-value passive parts, missing models, undefined parameters, and backend-specific warnings such as unsupported .control, .pz, or .sens usage.

Reusable Testbenches

The testbenches Python package provides reusable design recipes. They return DesignBench objects, which contain a normal pyspice_rs.Testbench plus intent, measurement names, result fields, and validation guidance.

import pyspice_rs as ps
from testbenches import amplifier_voltage_gain, ValidationRule, MetricSpec, extract_metrics, validate_metrics

dut = ps.Subcircuit("amp", ["vin", "vout"])
dut.R(name="bias", positive="vout", negative=dut.gnd, value=1e6)

bench = amplifier_voltage_gain(ps, dut, input_node="vin", output_node="vout")
print(bench.netlist("ngspice"))

metrics = {"gain_db": 40.1, "vref": 1.205}
report = validate_metrics(metrics, [
    ValidationRule("gain target", "gain_db", expected=40.0, tolerance=0.5),
])
print(report.passed)

Recipe families

  • Amplifiers: voltage gain, current gain, transimpedance, charge amplifier.
  • Converters: DAC static linearity and ADC ramp-code tests.
  • Switching: switch characterization, mux routing, demux routing, sample-and-hold.
  • Timing/RF: PLL lock and spur-oriented transient/Fourier setup.
  • References: bandgap reference line and temperature regulation setup.

Corners and Monte Carlo

from testbenches import CornerCase, MonteCarloPlan, corner_netlists, monte_carlo_netlist

corners = [
    CornerCase("tt_27", temperature=27, parameters={"vdd_nom": 3.3}),
    CornerCase("ss_hot", temperature=125, parameters={"vdd_nom": 3.0}),
]
netlists = corner_netlists(lambda: amplifier_voltage_gain(ps, dut), corners)

mc = MonteCarloPlan(samples=200, distributions={"Rload": "normal(1000,50)"})
xyce_netlist = monte_carlo_netlist(amplifier_voltage_gain(ps, dut), mc)

Validation helpers can parse CSV, TSV, whitespace scalar tables, repeated .MEASURE ... = value logs, Xyce scalar files such as .mt0/.ms0/.ma0, and Spectre scalar exports including mcdata with sibling mcparam labels.

Troubleshooting

ImportError: pyspice_rs not built

Run maturin develop in your active virtual environment. Tests skip Python bindings if the extension has not been built.

No backend found

Install a simulator and make sure its executable is on PATH. Start with ngspice unless your design specifically needs another backend.

Unsupported analysis

Try a backend that supports the requested analysis. For example, Xyce is used for statistical sampling and FFT metrics, Spectre for SpectreRF helpers, VacaSk or Spectre for stability, and ngspice for XSPICE and control-block workflows.

Model not found

Add model(...), include(...), or lib(...). Semiconductor and switch elements normally require simulator model definitions.

Unexpected node lookup failure

Use the exact node name emitted by the simulator. Result keys are backend-normalized where possible, but branch currents and special variables can still use simulator-specific names.
