Developer Notes

These notes are for maintaining the library. They summarize where code lives, how the pieces connect, and what to update when adding features.

Setup

nix develop
python3 -m venv .venv
source .venv/bin/activate
pip install maturin numpy pytest
maturin develop

Core checks:

cargo test
python3 -m pytest -v
cargo build --release --features cabi --no-default-features

Architecture

There are two user-facing paths.

  • Circuit is the direct Python-to-netlist builder.
  • Subcircuit and Testbench use the structured IR and backend-specific code generators.
Python API
  |
  |-- Circuit -----------------------> src/circuit.rs
  |
  |-- Subcircuit/Testbench ----------> src/ir/mod.rs
                                      |
                                      v
                               src/codegen/*
                                      |
                                      v
                               src/backend/*
                                      |
                                      v
                          rawfile/PSF parsers -> result wrappers

Prefer preserving structure until code generation. Avoid fixing backend syntax by rewriting already-emitted strings when a code generator can own the behavior.

Python Bindings

src/python.rs defines the public Python API. The package wrapper in python/pyspice_rs/__init__.py re-exports the native extension.

  • Use explicit #[pyo3(signature = ...)] annotations for user-facing methods.
  • Use PyValueArg for values that may be floats or unit values.
  • Raise ValueError for invalid input, KeyError for missing names, and RuntimeError for backend failures.
  • When adding a capability to Circuit, decide whether Subcircuit and Testbench need it too.

IR and Codegen

The IR in src/ir/mod.rs is the stable structured contract. Code generators turn it into dialect-specific text.

FileEmits
src/codegen/spice3.rsngspice, Xyce, and LTspice-style decks.
src/codegen/spectre.rsNative Spectre syntax.
src/codegen/vacask.rsVacaSk syntax.

When changing the IR, update the schema, Python conversion code, code generators, and tests together.

Backends

Backends implement invocation and capability reporting. The code generator owns netlist syntax; the backend driver owns executable discovery, subprocess/shared-library calls, output files, and parser selection.

Auto-selection filters backends by required features such as XSPICE, OSDI, measurements, step parameters, control blocks, Laplace sources, and Verilog co-simulation.

Results

Simulator output is normalized into result structs in src/result.rs. Rawfile parsing lives in src/rawfile.rs; Spectre PSF support lives in src/psf.rs.

When adding parser behavior, add fixtures for backend-specific formats. Do not assume every simulator names variables or measurements the same way.

Tests

Rust tests cover formatting, codegen, backend selection, simulation setup, and parsers. Python tests cover the public API, examples, linting, units, subcircuits, and testbenches.

cargo test
maturin develop
python3 -m pytest -v

Keep optional or commercial simulator tests skippable. For CI-friendly coverage, prefer netlist emission tests and parser fixtures.

Change Checklists

Add a circuit element

  1. Add the Rust element and display output in src/circuit.rs.
  2. Add an IR component if structured testbenches should support it.
  3. Update every relevant code generator.
  4. Expose the API in src/python.rs.
  5. Update feature detection if backend choice is affected.
  6. Add Rust and Python tests.

Add an analysis

  1. Add the simulation method in src/simulation.rs.
  2. Add an IR analysis variant if testbenches need it.
  3. Update codegen and backend routing.
  4. Add or reuse a result type.
  5. Expose the method in Python.
  6. Add tests and an example if the workflow is new.

Add a backend

  1. Add a module under src/backend/.
  2. Add BackendKind aliases, display name, and capabilities.
  3. Select or implement a code generator.
  4. Update detection.
  5. Add backend, codegen, and Python coverage.

Docs

  • docs/users.html: public user guide.
  • docs/developers.html: maintainer notes.
