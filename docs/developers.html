Developer Guide

Maintain, extend, test, and release DeSpice. This guide covers the Rust crate, PyO3 bindings, backend-neutral IR, code generators, simulator drivers, result parsing, linting, testbench helpers, C ABI, schemas, tests, and documentation publishing.

Architecture

DeSpice has two complementary paths: the classic Circuit builder for direct Python-to-netlist workflows, and the structured IR path used by Subcircuit/Testbench for backend-specific code generation.

Python user code
  |
  |-- Circuit API -------------------> src/circuit.rs ----------+
  |                                                            |
  |-- Subcircuit/Testbench API -----> src/ir/mod.rs --------+   |
                                                           |   |
                                                           v   v
                                                    src/codegen/* or legacy string deck
                                                           |
                                                    src/backend/* detection and run
                                                           |
                                                    simulator output files/stdout
                                                           |
                                                    src/rawfile.rs / src/psf.rs
                                                           |
                                                    src/result.rs normalized results
                                                           |
                                                    PyO3 result wrappers

The design goal is to keep semantics structured as long as possible. New backend-specific netlist behavior should usually be implemented in src/codegen/ rather than by string rewriting a previously emitted deck.

Python Bindings

src/python.rs defines the public Python API with PyO3. The wrapper package in python/pyspice_rs/__init__.py re-exports the native extension.

Major Python classes

Python class/functionRust wrapperRole
CircuitPyCircuitDirect circuit construction and simulation.
CircuitSimulatorPySimulatorRun analyses and configure saves, options, measurements, temperatures, steps, and backend.
SubcircuitPySubcircuitStructured reusable circuit block backed by IR.
TestbenchPyTestbenchDUT wrapper with stimulus, analyses, saves, corners, and netlist emission.
ModelLibraryPyModelLibraryPDK/model-library reference with backend-specific paths.
lintlint(...)Python entry point for src/lint.rs.
Unit, UnitValuePyUnit, PyUnitValuePython unit operator support.

Binding rules

  • Keep Python signatures explicit with #[pyo3(signature = ...)] so help text and errors stay predictable.
  • Map Python errors to specific exceptions: PyValueError for bad user input, PyKeyError for missing elements/nodes, PyRuntimeError for simulator/backend failures.
  • Use PyValueArg for user values that can be floats or unit values.
  • When adding an API to Circuit, consider whether Subcircuit and Testbench need the same capability.

Intermediate Representation

The IR in src/ir/mod.rs is the durable structured contract for backend-neutral circuits. It is serializable through Serde and mirrored by schema/circuit-ir.schema.json.

Core IR types

TypePurpose
CircuitIRComplete simulation document: top subcircuit, optional testbench, subcircuit definitions, model libraries.
SubcircuitNamed reusable circuit fragment with ports, parameters, components, instances, models, includes, libs, OSDI loads, and Verilog blocks.
ComponentTagged union for passive, source, semiconductor, switch, transmission line, XSPICE, and raw SPICE elements.
IrValueNumeric, expression, or raw value.
IrWaveformSin, pulse, PWL, exponential, SFFM, and AM source waveforms.
TestbenchDUT name, stimulus, analyses, options, saves, measures, temperature, initial conditions, node sets, step params, and extra lines.
AnalysisTagged union covering standard, RF, statistical, Xyce-specific, and Spectre-specific analyses.

When changing the IR

  1. Update src/ir/mod.rs and keep defaults backward compatible where practical.
  2. Update schema files under schema/.
  3. Update code generators for every backend that can represent the feature.
  4. Update Python conversion code in src/python.rs.
  5. Add Rust codegen tests and Python tests that serialize or emit the new field.

Code Generation

The CodeGen trait turns CircuitIR into backend-specific netlist text. Existing implementations include SPICE3 dialects, Spectre, and VacaSk.

FileResponsibility
src/codegen/spice3.rsSPICE3-style emission for ngspice, Xyce, and LTspice dialects.
src/codegen/spectre.rsNative Spectre syntax emission.
src/codegen/vacask.rsVacaSk-oriented syntax emission.
src/codegen/mod.rsTrait and shared entry points.

Prefer adding syntax decisions to the relevant code generator. Backend subprocess drivers should focus on invocation, temporary files, output paths, and parsing, not on semantic rewrites.

Backend System

src/backend/mod.rs defines the Backend trait, BackendKind, capability flags, analysis routing, feature-aware selection, and backend construction.

Backend trait

pub trait Backend: Send + Sync {
    fn name(&self) -> &str;
    fn run(&self, netlist: &str) -> Result<RawData, BackendError>;
    fn capabilities(&self) -> BackendCapabilities;
    fn codegen(&self) -> Box<dyn CodeGen>;
    fn run_netlist(&self, netlist: &str) -> Result<RawData, BackendError>;
}

Capability filtering

CircuitFeatures captures hard constraints: XSPICE, OSDI, measurements, step parameters, control blocks, Laplace sources, Verilog co-simulation, and element count. Auto-selection filters unavailable or incompatible backends before applying analysis preferences.

FeatureWhy it matters
has_xspiceOnly ngspice supports XSPICE A-elements.
has_osdiRequires backends that can load compiled Verilog-A/OSDI models.
has_measuresLimits selection to backends with measurement support.
has_step_paramsLimits selection to backends with step/sweep parameter support.
has_control_blocksOnly ngspice supports .control blocks in this project.
has_laplace_sourcesLaplace B-source support is backend-specific.
has_verilog_cosimRoutes to backends with digital co-simulation support.

Adding a backend

  1. Create a module in src/backend/ implementing Backend.
  2. Add a BackendKind variant, from_str aliases, display_name, and capabilities.
  3. Implement or select a CodeGen for the backend.
  4. Update detection in src/backend/detect.rs.
  5. Add analysis routing preferences if the backend should be preferred for specific analysis types.
  6. Add tests in tests/test_backends.rs, tests/test_codegen.rs, and Python tests if the backend is user-visible.
  7. Update user and developer documentation.

Result Parsing

Simulator output is normalized into result structs in src/result.rs. Rawfile parsing lives in src/rawfile.rs; Spectre PSF support lives in src/psf.rs.

Common result pattern

  • RawData stores title, plot name, variable metadata, real/complex data, and parsed measurements.
  • Analysis-specific structs such as OperatingPoint, DcAnalysis, AcAnalysis, and TransientAnalysis project raw data into ergonomic fields.
  • Python wrappers expose __getitem__, axis properties such as time/frequency/sweep, and measures.

When adding result fields, update both Rust structs and Python wrappers. Add parser fixtures for backend-specific output quirks rather than assuming all simulators emit identical raw data.

Linting

src/lint.rs provides static netlist checks used through the Python lint() function. The current test suite covers missing .end, missing ground, duplicates, floating nodes, zero values, missing models, undefined parameters, and backend-specific diagnostics.

New lint rules should return structured warning/error maps with stable fields so Python consumers can rely on them. Favor warning-level diagnostics for portability hazards and error-level diagnostics for conditions that are very likely to fail everywhere.

Testbench Helpers

The Python package testbenches is intentionally a thin recipe layer over pyspice_rs.Testbench, not a separate simulator framework.

FilePurpose
python/testbenches/design.pyDomain recipes returning DesignBench objects.
python/testbenches/analysis.pyValidation rules, metric reducers, corner evaluation, Monte Carlo plan emission, and scalar result parsers.
python/testbenches/__init__.pyPublic exports for users.

Recipe contract

  • Return a DesignBench with name, category, testbench, intent, measurements, result_fields, and validation.
  • Use ordinary Testbench stimulus calls such as V, I, waveform sources, R, and C.
  • Save every signal the validation layer needs.
  • Keep backend-specific behavior explicit through analysis methods, extra lines, CornerCase, or MonteCarloPlan.
  • Add tests that assert emitted netlists and serialized IR, not just object construction.

C ABI

The crate can be built with a C ABI for direct integration from C, C++, Zig, or other FFI consumers.

cargo build --release --features cabi --no-default-features

The cabi feature avoids the default Python feature set. Keep ABI changes conservative: exported symbol names, ownership rules, allocation/free pairing, and error reporting become external contracts once consumers link against them.

Testing

Rust tests

cargo test

Rust tests cover unit formatting, circuit display, code generation, backend selection/run-path behavior, simulation construction, and raw/result parsing.

Python tests

maturin develop
python3 -m pytest -v

Python tests cover the PyO3 API, units, circuits, netlists, examples, linting, subcircuits, testbenches, and design validation helpers. Example tests execute every script under examples/.

Focused test selection

python3 -m pytest tests/test_python_circuit.py -v
python3 -m pytest tests/test_design_testbenches.py -v
cargo test test_codegen

Simulator availability: Avoid making default tests require commercial or optional simulators unless the test can detect absence and skip cleanly. Netlist emission and parser fixture tests are usually better for CI portability.

Adding Features

Adding a circuit element

  1. Add the Rust element struct and Element variant in src/circuit.rs.
  2. Implement fmt::Display for legacy netlist output.
  3. Add an IR Component variant if the element should work in structured testbenches.
  4. Update code generators for SPICE3, Spectre, and VacaSk as applicable.
  5. Add PyO3 methods on Circuit, Subcircuit, and Testbench where appropriate.
  6. Update feature detection if the element constrains backend choice.
  7. Add tests for netlist output, IR serialization, backend-specific emission, and Python usage.

Adding an analysis

  1. Add the Rust method in src/simulation.rs.
  2. Add an IR Analysis variant and kind_str() mapping.
  3. Update codegen emission for testbenches.
  4. Update backend analysis preference routing.
  5. Create or reuse a result type in src/result.rs.
  6. Expose the method in src/python.rs on CircuitSimulator and Testbench.
  7. Add tests and documentation examples.

Adding a testbench recipe

  1. Add a recipe to python/testbenches/design.py.
  2. Export it from python/testbenches/__init__.py.
  3. Add tests asserting generated netlists, metadata, saved signals, measurements, and serialized IR.
  4. Add an example if it demonstrates a new workflow category.
  5. Document the recipe in the user guide.

Documentation Hosting

Documentation is now plain static HTML under docs/. GitHub Pages uploads that directory directly; there is no mdBook build step.

Editing docs

  • docs/index.html is the audience gateway.
  • docs/users.html is the public user manual.
  • docs/developers.html is the maintainer and contributor manual.
  • docs/styles.css contains shared visual styling.

Publishing workflow

The GitHub Actions workflow at .github/workflows/docs.yaml runs on pushes to main that touch docs/** or the workflow file, uploads docs/ as the Pages artifact, and deploys it with actions/deploy-pages.
